## Summary

- Scenarios are end-to-end acceptance tests that define what success looks like from the outside
- Each scenario is a markdown file with Setup, Action, and Expected sections
- Scenarios come before specs in the workflow: Task → Scenario → Spec → Prompts
- Scenarios accumulate — new features add scenarios, old ones never disappear
- Scenario files live in `scenarios/` with numeric prefixes for ordering

## Problem

Dark-factory had no way to describe what the system should do as a whole. Specs describe individual features. Prompts describe implementation steps. Nothing described the complete expected behavior from a user's perspective.

## Goal

A convention for writing end-to-end acceptance tests as markdown files that serve as both documentation and verification checklists.

## Desired Behavior

1. Scenarios live in `scenarios/` separate from specs and prompts.
2. Each scenario is a markdown file with Setup, Action, and Expected sections.
3. Scenarios use numeric prefixes for ordering (001, 002, ...).
4. Simpler scenarios come first — complex ones build on earlier ones.
5. Scenarios accumulate — never delete a passing scenario.
6. Scenarios test observable behavior (files, git state, CLI output), not internal implementation.
7. Planned scenarios use `status: idea` frontmatter.

## Scenario Format

```markdown
# Scenario NNN: Short description

What this scenario validates.

## Setup

Setup steps and checklist.

## Action

What to do.

## Expected

Observable outcomes as checklist items.

## Cleanup

Teardown steps.
```

## Constraints

- Scenarios are additive only — never delete or modify a passing scenario
- Scenario files are human-written, never generated by the implementation agent
- Scenarios test observable behavior, never internal implementation

## Acceptance Criteria

- [x] `scenarios/` directory exists
- [x] At least 3 scenarios covering direct workflow, PR workflow, and container smoke test
- [x] Scenario format documented
- [x] Planned scenarios tracked as `status: idea` files
